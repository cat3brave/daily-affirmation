# AGENTS.md

このファイルは、Daily Affirmation / 心のお守りアプリをCodexで編集するときの作業ルールです。

## 対象範囲

このルールは、このリポジトリ内のアプリ編集にだけ適用します。

## 基本方針

- 変更は小さな単位で行う
- 一度に大きく作り替えない
- 既存の世界観、UI、命名、コメントの雰囲気をできるだけ尊重する
- ユーザーが混乱しないように、作業後は変更内容を短く整理する
- 編集のたびにlintを実行せず、変更をまとめてから検証する

## Codex作業後の必須手順

コードまたは設定を修正したら、必ず以下を行う。

1. 修正がすべて完了するまで、編集のたびにlintを実行しない。
2. 複数ファイルを変更する場合も、変更をまとめてから検証する。
3. 修正完了後に `git status --short` を実行する。
4. `git diff --stat` を実行する。
5. `git diff --stat` は未追跡ファイルを含まないため、新規ファイルは `git status --short` で確認する。
6. コードまたは設定を変更した場合は、作業完了時に `npm.cmd run lint` を1回実行する。
7. コードまたはテストを変更した場合は、作業完了時に `npm.cmd run test` を1回実行する。
8. lint成功後にコードや設定を変更していない場合は、同じlintを再実行しない。
9. test成功後にコードやテストを変更していない場合は、同じtestを再実行しない。
10. lintエラーの原因調査や修正確認が必要な場合に限り、変更ファイル単位で `npx eslint <changed files>` を実行してよい。
11. `npm.cmd run lint` が120秒でタイムアウトした場合は、無限に再実行しない。
12. 全体lintがタイムアウトした場合は、必要に応じて変更ファイル単位のeslintを実行し、変更ファイルの確認結果と全体lintがタイムアウトした事実を報告する。
13. ドキュメントのみの変更ではlintとtestを省略してよい。
14. 変更内容と確認結果を短く要約する。

## 報告形式

作業後は、以下の形式で報告する。

```txt
実行したこと:
-

確認結果:
- npm.cmd run lint: 成功 / 失敗 / タイムアウト / 省略
- npm.cmd run test: 成功 / 失敗 / タイムアウト / 省略
- git status --short:
- git diff --stat:

変更要約:
-

次に確認してほしいこと:
-
```

## commit / push について

- ユーザーが明示的に依頼するまで、`git add`、`git commit`、`git push` は実行しない
- commitを依頼された場合も、先に `npm.cmd run lint` と `npm.cmd run test` が成功していることを確認する
- lint成功後にコードまたは設定を変更した場合は、commit前にもう一度 `npm.cmd run lint` を実行する
- test成功後にコードまたはテストを変更した場合は、commit前にもう一度 `npm.cmd run test` を実行する
- `git add .` はなるべく使わず、変更対象ファイルを指定する
- `.env.local` や秘密情報を含むファイルは絶対にcommitしない

## エラー時のルール

- `npm.cmd run lint` が失敗した場合は、commitやpushをせずに停止する
- `npm.cmd run test` が失敗した場合は、commitやpushをせずに停止する
- lintエラーを修正した場合は、修正完了後に `npm.cmd run lint` を1回だけ再実行する
- testエラーを修正した場合は、修正完了後に `npm.cmd run test` を1回だけ再実行する
- コードを変更せずに、同じ失敗コマンドを繰り返し実行しない
- エラー内容を短く正確に要約して報告する
- 原因が不明な場合は、推測で断定しない
- タイムアウトとlintエラーを区別して報告する
